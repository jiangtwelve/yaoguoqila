# TASK-009: 设计并实现全页面下拉刷新

## Objective
为首页、物品详情页等需要主动同步数据的页面设计统一的下拉刷新交互，使用户可以主动同步页面数据，同时保持当前「流光毛玻璃」视觉风格。

## Interaction Design Draft

> 用户已确认该交互方向。实现仍按总计划排在 TASK-008 后执行。

### 全局规则
- 首页、物品详情页启用下拉刷新；新增/编辑物品页不启用下拉刷新。
- 下拉刷新只刷新当前页面必要数据，不改变用户已输入但未保存的表单内容。
- 刷新期间使用系统下拉刷新手势为主，页面内不额外堆叠大型 loading 卡片。
- 刷新完成后不弹“已刷新”提示，使用原生下拉区域 loading 表达刷新中；失败时显示可读错误提示，不清空已有页面内容。
- 刷新不应破坏当前页面上下文：详情页仍停留当前物品，首页尽量保留滚动位置。

### 首页
- 下拉动作：重新拉取当前家庭、物品列表、位置和统计相关数据。
- 刷新中：保持当前列表可见，不切换到整页 loading。
- 刷新完成：
  - 如果没有搜索词，更新完整列表并尽量保留原滚动位置。
  - 如果有搜索词，刷新后继续应用当前搜索词。
  - 如果某个物品因删除、用完或搜索结果变化消失，不强制跳转。
- 失败：保留旧列表，提示“刷新失败，稍后再试”。

### 物品详情页
- 下拉动作：重新获取当前物品详情。
- 刷新中：保留当前详情内容和图片走马灯位置，不切换成空白加载态。
- 刷新完成：更新状态、图片、到期日、位置、备注等字段。
- 如果物品已被删除或无权限：显示错误状态，并提供返回首页入口。

### 新增/编辑物品页
- 用户确认新增物品页面不需要下拉刷新。
- 新增/编辑物品页不启用下拉刷新，避免表单录入过程中出现无意义操作或误以为会同步未保存内容。

### 视觉与动效
- 使用小程序原生下拉刷新能力优先，避免自绘复杂刷新组件。
- 刷新 loading 只出现在原生下拉区域，完成后下拉区域再回收。
- 颜色使用当前主题色 `#4f7f78`。
- 文案保持轻量，不使用夸张动效。
- 不在页面顶部新增固定刷新按钮，避免和微信胶囊操作栏冲突。

## Scope
- 开启页面级下拉刷新配置。
- 为首页、物品详情页实现 `onPullDownRefresh`。
- 抽取必要的刷新 helper，保证刷新完成后调用 `uni.stopPullDownRefresh()`。
- 处理刷新成功、失败和下拉 loading 最短停留时长。

## Non-Scope
- 不做自定义刷新动画。
- 不做离线缓存同步。
- 不做后台静默自动刷新策略。
- 不覆盖用户未保存的表单输入。

## Acceptance Criteria
- 用户确认首页、详情页下拉刷新行为符合预期。
- 刷新不会导致首页无故回到顶部。
- 新增/编辑物品页不启用下拉刷新。
- 刷新失败时不会清空已有页面内容。

## Implementation Notes
- `src/pages.json` 已为首页、物品详情页开启 `enablePullDownRefresh`。
- 首页实现 `onPullDownRefresh`：刷新当前家庭与物品列表，保留当前搜索词，不切换到整页 loading，并关闭已打开的左滑删除行。
- 详情页实现 `onPullDownRefresh`：刷新当前物品详情，保留当前页面内容，不切换到空白加载态。
- 新增/编辑页不启用下拉刷新。
- 刷新完成统一调用 `uni.stopPullDownRefresh()`；成功后不弹窗提示，失败时才给轻量 toast 反馈。
- 新增 `waitForMinimumRefreshIndicator()` 和 `waitForFailedRefreshIndicator()` 控制顶部小浮层的最短展示时长，避免 mock 数据快速返回导致状态一闪而过。
- 排查发现自定义导航和首页根容器 `overflow: hidden` 会让原生下拉 loading 可视区域过小；已移除首页根容器溢出限制，并在首页/详情页增加页面内轻量刷新指示器，确保下拉触发后可见。
- 根据用户反馈，原生下拉区域现在触发后立即收回；刷新状态由顶部小浮层承接。刷新中显示“正在刷新”，失败时同一个浮层切换为“刷新失败，稍后再试”后自动收起。
